# System Architecture Overview

SveltyCMS is built with the **"High-Performance Edge"** philosophy: zero-runtime overhead, sub-1ms API hot-paths, and the **Self-Healing State Machine** for automated recovery.

---

## 🏗️ The Core Engine: Svelte 5 & Runes

SveltyCMS leverages **Svelte 5** to provide fine-grained reactivity without the "Virtual DOM" tax of React-based competitors.

### 👥 For Administrators
The Admin Studio is a **single-page application (SPA)** with server-side rendering (SSR) for the initial load. It uses:
- `$state()` — Deeply reactive state management for real-time field edits.
- `$derived()` — Instant UI computations (e.g., search results, media thumbnails).
- `$effect()` — Managed side-effects for API syncing.

## 🗄️ Database Agnosticism: The Adapter Pattern

SveltyCMS is not tied to any specific database. It uses a **unified CRUD interface** that translates operations into native driver calls.

### 🛠️ For Digital Engineers
The architecture relies on the `DatabaseAdapter` interface, which is physically separated from your application logic.

## 🧩 Modularity: The Widget Strategy

SveltyCMS is **modular by design**. Every field in a collection is a **Widget**.

### 🧩 3-Pillar Architecture
1.  **Definition**: Valibot-based validation schemas.
2.  **Input**: The Svelte 5 edit component.
3.  **Display**: The read-only render component.

## 🧪 System Health: The State Machine

SveltyCMS uses a custom state machine to manage its lifecycle.

- `IDLE` — Initial state; waiting for setup completion or triggering initialization.
- `INITIALIZING` — Database connection and core services are starting up.
- `READY` — Normal operational state. All systems go.
- `DEGRADED` — Operational but some non-critical services are failing.
- `FAILED` — Critical path error (automated recovery mode triggers).
