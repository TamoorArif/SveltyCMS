# Security Testing

**Last Updated:** March 13, 2026

SveltyCMS implements rigorous security testing to ensure the highest level of protection for user data and system integrity. All security tests follow the **no-backdoor policy** - tests must pass through real security guards without any mocked permission checks or test bypass mechanisms.

**Test Files:**

- [`tests/unit/auth-lockout.test.ts`](../../tests/unit/auth-lockout.test.ts) - Auth security features

**Coverage:**

- Account blocking/unblocking

- Session extension on activity
- Failed attempt tracking
- Security event logging

### 3. No Backdoor Policy

All security tests adhere to strict no-backdoor principles:

- ✅ **No test API keys** - Tests use real authentication flows
- ✅ **No bypass headers** - All requests go through standard auth
- ✅ **No mocked permissions** - RBAC tested with real role checks
- ✅ **No shortcuts** - Encryption tested with real key derivation

## Running Security Tests

```bash
# Run all security tests
bun test tests/unit/utils/crypto.test.ts
bun test tests/unit/auth-lockout.test.ts

# Run security tests with coverage
bun test --coverage tests/unit/security/

# Integration security tests (requires running server)
bun run test:integration --grep "security"
```

## Security Test Architecture

### White-Box Testing (Unit)

Security unit tests verify:

- Cryptographic primitives work correctly
- Authentication logic handles edge cases
- Session management enforces policies
- Input validation rejects malicious data

### Black-Box Testing (Integration)

Security integration tests verify:

- API endpoints enforce authentication
- RBAC blocks unauthorized access
- Rate limiting prevents brute force
- Firewalls filter malicious requests
