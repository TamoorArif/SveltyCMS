### Unit Test Architecture (Vitest 4.1.0)

Our unit tests leverage **Vitest** for native Svelte 5 rune support and fast execution:

- **Dedicated Vitest Config**: `vitest.config.ts` manages all aliases (`@widgets`, `@services`, `$paraglide`).
- **SvelteKit Mocks**: Specialized mock files in `tests/unit/mocks/$app/` simulate the SvelteKit environment without the full runtime overhead.
- **Rune Stabilization**: Fixed recursive update loops in `UIStore` by moving to granular property updates.
- **Isomorphic Mocking**: `tests/unit/setup.ts` provides a consistent environment for both Browser and Node-based logic.

### Automated CI/CD

Our GitHub Actions pipeline is 100% synchronized with the local environment:

- **Vite 8 Compatibility**: Optimized for the latest Vite ecosystem.
- **Timer Stabilization**: `TEST_MODE` now bypasses non-deterministic timers in `LoadingStore` and `ToastStore` for synchronous reliability.
- **Matrix Testing**: Integration and E2E tests are executed against all supported databases (MongoDB, MariaDB, PostgreSQL, SQLite).

## CI Workflow & Data Isolation (Latest)

We have transitioned to a **Pure Black-Box** CI strategy to ensure 100% data isolation and environment parity:

- **No Fake Configs**: Manual creation of `private.test.ts` in CI is removed.
- **Native Setup**: CI jobs now run `scripts/setup-system.ts` against a live server to generate real test configurations.
- **Cache-Busting Imports**: Dynamic imports of configuration files now use a timestamp query parameter to ensure in-process reinitialization picks up disk changes immediately.
- **Server Resilience**: E2E jobs include explicit server restarts and health checks after setup to guarantee a fresh, verified state.
- **Unit Isolation**: Unit tests remain 100% isolated via in-memory mocks, completely decoupled from the physical filesystem.
