# Performance Benchmarks

SveltyCMS is designed for sub-millisecond persistence and high-concurrency GraphQL delivery. This document outlines our benchmarking methodology and current baseline results.

## Benchmarking Environment

All performance metrics were gathered on the following hardware:

- **Hardware**: AMD Ryzen 5 3600 (Reference Point)
- **CPU**: 6-core, 12-thread (Shared Resources)
- **RAM**: 64 GB
- **OS**: Ubuntu 24.04
- **Runtime**: Node.js 25.8.1 / Bun 1.3.11
- **Tooling**: Vite+ Core Engine

## Hardware Context & Comparison

All tests run on a dedicated server:

- AMD Ryzen 5 3600 (6C/12T, 2019 Zen 2)
- 64 GB DDR4 RAM
- Dual 512 GB NVMe SSD (RAID 1)

### The GraphQL JIT Engine

With the introduction of `@envelop/graphql-jit`, the GraphQL benchmark suite now resolves operations via dynamically compiled JavaScript functions in memory instead of constantly re-evaluating the AST. This significantly lowers GC overhead and accelerates deeply nested queries.

## Multi-Database Performance Comparison

SveltyCMS is database-agnostic. While performance varies by engine and storage medium (NVMe vs SATA), our [Unified Caching Layer](../docs/database/index.mdx#unified-caching-flow) ensures sub-millisecond response times for hot paths across all adapters.

### Steady-State Latency (Server-Native)

| Operation              | MongoDB (Raw) | MariaDB (SQL) | PostgreSQL (SQL) | SQLite (Local) |
| :--------------------- | :------------ | :------------ | :--------------- | :------------- |
| User lookup (cached)   | 0.8 ms        | 0.6 ms        | 0.7 ms           | 0.2 ms         |
| User lookup (uncached) | 2.0 ms        | 5.8 ms        | 3.0 ms           | 4.5 ms         |
| Collection list (1K)   | 18.0 ms       | 8.5 ms        | 4.2 ms           | 1.2 ms         |
| Content tree (500)     | 15.0 ms       | 4.8 ms        | 3.5 ms           | 1.2 ms         |
| Dashboard (cached)     | 5.0 ms        | 6.0 ms        | 8.0 ms           | 4.0 ms         |

### Optimization Impact (Enterprise Scale)

Measured on a collection with 100,000 entries and 35 fields:

| Metric                             | Before     | After     | Factor    |
| :--------------------------------- | :--------- | :-------- | :-------- |
| Steady-state DB operation latency  | ~6–10 ms   | ~2.75 ms  | ~2.5–3.6× |
| Cold-start time (models + connect) | ~1–2 s     | ~138 ms   | ~7–14×    |
| Cached dashboard / list payload    | 450–890 KB | 95–180 KB | 79–86% ↓  |
| First content load (uncached)      | —          | < 5 ms    | —         |
| Session validation                 | ~45 ms     | ~0.8 ms   | ~56×      |
| **p99 tail latency**               | ~180 ms    | ~18 ms    | **10×**   |

> [!NOTE]
> Performance gains represent a **95–99% reduction** in latency across typical user flows compared to unoptimized MongoDB/SQL queries. This translates to 20–160× faster operations depending on cache hit rate and query complexity.

| Operation                 | Current Latency (ms) | Baseline (Target) | Delta (%) |
| :------------------------ | :------------------- | :---------------- | :-------- |
| **Document Write/Insert** | **35.26 ms**         | 35.26 ms          | 0.0%      |
| **Document Read/PK**      | **0.13 ms**          | 0.13 ms           | 0.0%      |
| **Document Update**       | **34.02 ms**         | 34.02 ms          | 0.0%      |
| **Document Delete**       | **33.98 ms**         | 33.98 ms          | 0.0%      |

> [!NOTE]
> Write latencies reflect real-world server conditions (Plesk-managed Ubuntu 24.04, software RAID-1 NVMe SSD, full security stack). In optimized local/dev environments with direct NVMe access and tuned WAL PRAGMAs, writes often drop below 5 ms.

#### Understanding the Metrics:

- **Baseline (Target)**: These values represent the **Server-Native Baseline** established on our AMD Ryzen 5 3600 (Ubuntu/Plesk) environment. All future performance changes will be measured against this standard.
- **Delta (%)**: The relative difference between current performance and the server-native baseline.
- **The "0% Delta" Explained**: We have standardized our benchmarks to reflect the actual hosting environment (shared SATA/HDD storage), moving away from notebook-based (NVMe) targets.
- **Goal**: While the baseline reflects current hardware limits, our goal remains to optimize persistence logic to achieve the lowest possible latency on this infrastructure.

## Deeply Nested Queries

SveltyCMS supports efficient resolution of circular and deeply nested relationships.

### Example Nested Query:

```graphql
query NestedRelation {
  Authors {
    name
    Posts {
      title
      Author {
        name
      }
    }
  }
}
```

This query resolves through the `Relation` widget which uses the optimized `dbAdapter` context to batch-fetch related documents, minimizing N+1 query issues.

## Running the Benchmarks

To execute the full performance suite:

```bash
# Start the production/preview server
bun run build && bun run preview

# Run the API Benchmarking Suite
bun run test:bench:all

# Run specific API stress tests
bun run test:bench:graphql
bun run test:bench:rest

# Alternatively, manual load levels
LOAD_LEVEL=EXTREME bun run tests/benchmarks/graphql-stress.ts
```

For database-specific benchmarks:

```bash
bun run tests/benchmarks/database-performance.ts
```

## Caveats & Future Work

- **Query Complexity**: Benchmarks currently use simple-to-medium queries; deeply nested relational queries may show higher latency.
- **Environment**: Results are server-specific; cloud/edge deployments (e.g., Vercel Edge, Cloudflare Workers) would likely show different profiles.
- **Upcoming**:
  - Cursor-based pagination stress tests.
  - Comprehensive multi-DB comparison (MongoDB vs PostgreSQL vs SQLite).
  - Hardened RBAC impact benchmarks.
- **Goal**: Sustain >1,000 req/sec with p99 < 100 ms on this hardware with increasingly complex workloads.
