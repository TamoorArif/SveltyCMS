# SAML 2.0 / Enterprise SSO API

SveltyCMS provides SAML 2.0 integration for enterprise identity providers like Okta, Azure AD (Entra ID), and PingIdentity. This is built on **BoxyHQ SAML Jackson** and supports multi-tenant isolation with Just-In-Time (JIT) user provisioning.

## Endpoints

### 1. Configure SAML Connection

**Endpoint**: `POST /api/auth/saml/config`  
**Auth**: Required (Admin role)

Registers or updates a SAML 2.0 connection for a specific tenant by providing the IdP's XML metadata.

**Payload**:

| Field | Type | Description |
| :---- | :--- | :---------- |
| `tenant` | `string` | The tenant identifier this connection is scoped to. |
| `product` | `string` | Product identifier — use `sveltycms`. |
| `rawMetadata` | `string` | The full XML metadata document from your IdP. |
| `defaultRedirectUrl` | `string` | URL to redirect to after a successful SSO authentication. |

```json
{
  "tenant": "my-tenant",
  "product": "sveltycms",
  "rawMetadata": "<EntityDescriptor ...>...</EntityDescriptor>",
  "defaultRedirectUrl": "https://cms.example.com/admin"
}
```

> [!NOTE]
> In multi-tenant mode, admins may only register SAML connections for their own `tenantId`. Super-admins may configure connections for any tenant.

### 2. Initiate SAML Login

**GET** `/api/auth/saml/login`

Initiates the SAML SSO flow by redirecting the user to their configured Identity Provider.

**Authentication**: Public

**Query Parameters**:

- `tenant` (string): Required. The tenant identifier.
- `product` (string): Optional. Defaults to `sveltycms`.

**Example**: `GET /api/auth/saml/login?tenant=my-tenant`

### 3. Assertion Consumer Service (ACS)

**POST** `/api/auth/saml/acs`

The callback endpoint where the IdP sends the SAML assertion.

**Authentication**: Public (IdP Callback)

**Body**: Multipart/Form-Data (SAMLResponse, RelayState)

**Features**:

- **Just-In-Time (JIT) Provisioning**: Automatically creates users on first login if enabled.
- **Tenant Isolation**: Strictly enforces that users are logged into the correct tenant.
- **Security**: Validates signatures, audiences, and expirations.

## 4. Multi-Tenant & Just-In-Time Provisioning

### Multi-Tenant Considerations

When `MULTI_TENANT` is enabled:

- Admins can only configure SAML for their own `tenantId`.
- `GET /api/auth/saml/login` requests **must** include the `tenant` query parameter.
- The ACS endpoint validates that the assertion's audience and recipient belong to the requested tenant before creating a session.

### Just-In-Time (JIT) Provisioning

If `SAML_JIT_PROVISIONING` is enabled in system settings, user accounts are created automatically on first successful login:

- New users receive the default role `VIEWER` (configurable per tenant).
- Users are automatically associated with the authenticated `tenantId`.
- Subsequent logins link to the existing account without creating duplicates.
